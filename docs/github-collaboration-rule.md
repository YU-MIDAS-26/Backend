# GitHub 협업 규칙

본 프로젝트는 `main` 브랜치와 팀원별 개인 개발 브랜치 구조로 협업을 진행한다.

## 1. 브랜치 운영 방식

- `main` 브랜치는 최종 통합 브랜치로 사용한다.
- 팀원은 각자 개인 개발 브랜치에서 작업을 진행한다.
- 모든 작업 결과는 Pull Request를 통해 `main` 브랜치에 반영한다.
- `main` 브랜치에는 직접 push하지 않는다.

예시:
- `main`
- `woohyeon`
- `yujin`
- `minhyung`

---

## 2. main 브랜치 규칙

`main` 브랜치는 저장소의 최종 통합 브랜치이므로 아래 규칙을 적용한다.

- 직접 push를 금지한다.
- Pull Request를 통해서만 병합할 수 있도록 한다.
- Pull Request 병합 전 최소 1회의 승인(Approve)을 요구한다.
- 리뷰 중 생성된 대화(Conversation)가 모두 해결된 경우에만 병합할 수 있도록 한다.
- force push를 금지한다.
- 브랜치 삭제를 금지한다.
- 병합 방식은 `Squash merge`를 기본으로 한다.
- 관리자는 기본적으로 동일한 규칙을 따르되, 필요한 경우 우회 권한을 통해 병합할 수 있도록 한다.

---

## 3. 개인 브랜치 규칙

- 팀원은 각자 자신의 브랜치에서 자유롭게 작업할 수 있다.
- 기능 구현, 수정, 실험 등은 모두 개인 브랜치에서 진행한다.
- 개인 브랜치 작업 완료 후 `main` 브랜치로 Pull Request를 생성한다.

---

## 4. Pull Request 규칙

- Pull Request 제목과 내용은 작업 내용을 명확히 드러내도록 작성한다.
- 승인 없이 임의로 병합하지 않는다.
- 리뷰에서 요청된 수정 사항이 있으면 반영 후 다시 확인을 받는다.
- 병합 전 대화가 모두 해결되었는지 확인한다.
- 최종 병합은 관리자만 수행한다.

---

## 5. 병합 방식

- `main` 브랜치 병합은 `Squash merge`를 사용한다.
- 이를 통해 불필요하게 많은 커밋이 `main` 브랜치에 직접 누적되지 않도록 한다.
- 하나의 Pull Request는 하나의 정리된 커밋으로 반영하는 것을 원칙으로 한다.
